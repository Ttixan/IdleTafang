# 开发计划（塔防游戏 MVP）

## 1. 目标

- 在 1–2 个月内完成一个可玩的 MVP。
- 验证核心循环：工作阶段打字产能 → 休息阶段塔防防守 → 结算与成长回流。
- 保证后续可扩展、可测试、可维护。

## 2. 开发原则

- 先闭环，再扩展。
- 先规则，再表现。
- 先配置化，再硬编码补齐。
- 先做能玩版本，再做平衡和打磨。

## 3. 里程碑总览

### 第 1 阶段：项目骨架
- 搭建 Unity 工程结构。
- 建立场景流转与状态机。
- 接入输入、计时器、事件机制。
- 完成最小运行框架。

### 第 2 阶段：核心玩法闭环
- 实现打字产能系统。
- 实现基础资源系统。
- 实现基础塔、防御路径、敌人和波次。
- 打通战斗与结算。

### 第 3 阶段：局外成长与存档
- 实现局外成长。
- 实现本地存档。
- 打通跨局资源回流。
- 完成基础 UI 串联。

### 第 4 阶段：平衡与验证
- 进行数值平衡。
- 补齐测试。
- 修复流程问题。
- 形成首轮可体验版本。

## 4. 周计划

### 第 1 周：骨架搭建
- 创建 `Boot`、`MainMenu`、`Run` 场景。
- 完成 `GameLoop` 状态机。
- 建立 `Core / Gameplay / UI / Config` 目录结构。
- 接入基础输入与计时器。

### 第 2 周：打字系统
- 实现输入采集。
- 实现有效字符判定。
- 实现准确率、连击、节奏分数。
- 将打字结果映射为能量或金币。

### 第 3 周：经济与战斗基础
- 实现资源收支系统。
- 实现建造与升级的最小流程。
- 实现 1 条固定路径。
- 实现 1 种塔和 1 种敌人。

### 第 4 周：波次与结算
- 实现波次配置读取。
- 实现刷怪与结束判定。
- 完成战斗胜负结算。
- 完成奖励发放与回流。

### 第 5 周：局外成长
- 实现局外成长节点。
- 实现本地存档读写。
- 完成启动时加载与退出时保存。
- 增加基础设置项。

### 第 6 周：UI 串联
- 完成 HUD。
- 完成阶段面板。
- 完成结算面板。
- 串联主菜单到运行流程。

### 第 7 周：平衡调整
- 调整产能、消耗和奖励曲线。
- 调整塔、敌人、波次参数。
- 观察单局时长与循环节奏。
- 修复明显的卡点或失衡点。

### 第 8 周：验证与收尾
- 进行 EditMode / PlayMode 基础测试。
- 修复存档、阶段切换、结算问题。
- 补充说明文档。
- 整理首版发布清单。

## 5. 任务拆分

### P0：必须完成
- 状态机。
- 打字系统最小实现。
- 资源系统。
- 1 条路径、1 种塔、1 种敌人。
- 波次与结算闭环。
- 存档。

### P1：尽量完成
- 基础 UI。
- 局外成长。
- 配置导入。
- 基础测试。

### P2：后续再做
- 更多塔和敌人。
- 更复杂的局外成长。
- 更丰富的视觉表现。
- 更完整的新手引导。

## 6. 验收标准

### 可玩性
- 玩家能够从主菜单进入一局。
- 玩家能够完成工作阶段与休息阶段循环。
- 玩家能够完成至少一波战斗并获得结算。

### 系统性
- 资源变化可追踪。
- 存档可读写。
- 配置可调整。
- 核心逻辑可测试。

### 体验性
- 反馈清晰。
- 流程不断裂。
- 首版不会频繁崩溃或卡死。

## 7. 风险点

- 打字与战斗割裂，导致核心循环不连贯。
- 塔防内容扩展过快，拖慢 MVP 进度。
- 数值过早失衡，导致首轮体验过难或过轻。
- 存档与配置格式过早复杂化，增加维护成本。

## 8. 当前建议

- 先按第 1–4 周做出能玩的闭环。
- 第 5–8 周再做成长、UI 和平衡。
- 所有新增内容都优先服务于“能玩一局”的目标。

## 9. 当前执行重点

- 先补纯逻辑层测试，覆盖打字统计、资源钱包、状态机切换。
- 优先验证不依赖 Unity 场景的核心规则，保证后续扩展不会破坏闭环。
- 等纯逻辑稳定后，再继续补 UI 和战斗流程的联调测试。
