# 技术方案与框架设计（Unity + C#）

## 1. 目标

本阶段目标是做一个**极简可玩 Demo**，优先跑通闭环：
- 打字产能
- 能量建造/升级
- 刷怪与波次推进（敌人从四周刷出并向中心移动）
- 基础 UI 显示与结算

Unity 只负责：
- 渲染
- 输入
- UI
- 场景承载

核心规则尽量保持纯 C#。

## 2. 技术选择

- Unity：2022.3 LTS
- 语言：C#
- 渲染：3D 俯视角
- UI：UGUI + TextMeshPro
- 配置：MVP 阶段优先常量/脚本，后续再配置化
- 存档：后续阶段实现

## 3. 架构原则

- 逻辑优先，表现其次。
- Unity 只做展示和桥接。
- 先做“少代码、少绑定、少对象”的可玩版本，再逐步工程化。

## 4. 分层与职责（当前实现对照）

### 4.3 UI

## 5. 当前数据流（最小闭环）

```text
Keyboard Input
  -> TypingInputRouter
  -> TypingSession
  -> TypingRewardCalculator
  -> ResourceWallet
  -> BuildService
  -> UI (RunHudController / BuildPanelView / HudView)

CombatWaveManagerLogic (纯逻辑，决定刷怪节奏)
  -> CombatWaveManager (Unity 适配：Instantiate prefab、事件上报、对象生命周期)
  -> CombatEnemy (Unity 适配：移动/到达事件/销毁)
  -> UI (CombatStatusPanelView)
```

## 6. 场景结构

- `Boot.unity`
- `MainMenu.unity`
- `Run.unity`

约定：`Run` 场景尽量保持少对象，只保留最少可运行内容；UI 结构在 Hierarchy 可见，避免运行时动态创建 UI。

## 7. 资产与接线指南（战斗资产最小集）

### 7.1 接线原则
- 纯逻辑只负责规则、状态、位置计算。
- Unity 组件只负责读取场景对象、实例化 prefab、同步 Transform。
- 场景里不要把规则写到 MonoBehaviour 里；prefab 只承载表现与引用。

### 7.2 必做 Prefab / 场景对象
1. `CombatEnemy` prefab
   - 挂 `CombatEnemy`
   - 有可见模型（或简单几何体）
2. `CombatArena`
   - 挂 `CombatArena`
   - 包含 `CenterPoint` 子物体
3. `CombatPath`
   - 挂 `CombatPath`
   - 包含若干 `CombatSpawnPoint` 子物体（表示刷怪点）
4. `HUD`
   - `RunHudController` 绑定 `BuildPanelView`、`HudView`、`CombatStatusPanelView` 与结算面板引用

### 7.3 推荐的 Run 场景 Hierarchy（示例）

```text
Run
├── CombatArena
│   └── CenterPoint
├── CombatPath
│   ├── SpawnPoint_0
│   ├── SpawnPoint_1
│   └── SpawnPoint_2
├── WaveManager (CombatWaveManager)
├── TypingInputRouter
└── Canvas
    ├── HUD (RunHudController)
    ├── BuildPanel
    ├── CombatStatusPanel
    └── GameOverPanel (结算面板：Title / Summary / Retry / MainMenu)
```

### 7.4 最小完成标准（用于接线自检）
- 能从主菜单进入 `Run`。
- HUD 能显示能量/金币等关键资源。
- 输入能产能；升级能消耗能量并提升等级。
- 战斗能刷出敌人并朝中心移动。
- 至少一波能跑完并能触发胜负结算。
