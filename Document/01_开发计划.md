# 开发计划（塔防游戏 MVP）

## 1. 目标

- 在 1–2 个月内完成一个可玩的 MVP。
- 验证核心循环：工作阶段打字产能 → 休息阶段塔防防守 → 结算与成长回流。
- 保证后续可扩展、可测试、可维护。

## 2. 开发原则

- 先闭环，再扩展。
- 先规则，再表现。
- 先配置化，再硬编码补齐。
- 先做能玩版本，再做平衡和打磨。

## 4.1 P1 玩法增强（详细计划）

> 目标：让玩家“看得见、升级有意义、战斗中有可操作的事做”，同时不破坏现有 MVP 闭环。

### P1 总体验目标（验收口径）
- [x] 信息可见：玩家随时知道能量/金币的获取与消耗、塔的等级与效果变化（含可选 `CombatStatsText` / Debug）。
- [x] 升级有意义：升级带来可量化的战斗收益（基地最大血量 + 炮台伤害/射速，见 `BuildPrototype`）。
- [x] 等待不无聊：波次进行时提供可操作且能影响结果的玩法（`ManualTurretController` + 准星可选）。

### P1-1 能量/金币 UI 可见（打字积攒、升级消耗）
#### 目标
- HUD 固定显示 `Energy` 与 `Gold`。
- 建造/升级面板显示花费，并在资源不足时禁用按钮或提示原因。

#### 验收标准
- 打字提交后能量数值在 UI 立即更新。
- 点击升级后能量立即扣除；能量不足时升级不会发生且按钮不可交互（或有提示）。
- 不开启 DebugText 也能理解资源循环与消耗逻辑。

#### 任务拆分
- [x] T1：扩展 `HudView`：增加能量/金币文本引用与 `SetEnergy(int)` / `SetGold(int)`。
- [x] T2：在 `RunHudController` 的关键事件点刷新 HUD（打字提交、升级成功/失败、结算）。
- [x] T3：扩展 `BuildPanelView`：显示 cost，并根据 `wallet.Energy >= cost` 控制升级按钮可用性。

### P1-2 升级产生实质效果（先做“立刻可感知”的提升）
#### 设计原则
- P1 优先做“能立刻看见差异”的提升：升级影响基地耐久与手动炮台强度。
- 自动塔（持续输出）属于下一步扩展：需要先补齐“塔对敌人攻击”的基础逻辑。

#### 建议最小效果集（P1 版）
- 基地最大血量：每级提升（例：+1/+2）。
- 手动炮台：伤害与射速随等级提升（例：每级 +1 伤害，冷却略减少）。

#### 验收标准
- 同样一波配置下，升级前后基地掉血/逃逸数差异明显，或清怪效率明显改善。
- UI 能清楚显示等级与关键属性变化（至少展示等级与一个关键属性的变化结果）。

#### 任务拆分
- [x] T4：扩展 `BuildPrototype`：提供等级驱动的属性接口（如 BaseHPBonus / TurretDamage / TurretCooldown）。
- [x] T5：扩展战斗侧接口以接收升级效果（如 `CombatWaveManager.ApplyMaxBaseHealth(int newMax, ...)` 等），确保升级能影响战斗。

### P1-3 手动炮台：瞄准与打击敌人（让等待波次也有事做）
#### 最小玩法定义
- 鼠标/准星瞄准，按键/点击射击（射线或弹丸）。
- 有射速冷却，避免无限连点无脑刷。
- 伤害与射速至少受“升级等级”影响，形成闭环。

#### 实现路径（低侵入）
- 新增 `ManualTurretController`（MonoBehaviour）：处理输入、射击冷却、命中判定与伤害。
- 扩展 `CombatEnemy`：增加 `HP` 与 `TakeDamage(int)`；死亡时由 `CombatWaveManager` 正确移除并销毁对象。

#### 验收标准
- 玩家在战斗中可通过炮台操作明显影响结果（减少逃逸/延缓失败）。
- 敌人死亡/销毁链路稳定：事件解绑干净，列表清理正确，不产生残留对象。
- 玩法反馈明确（至少命中能看见敌人血量减少或敌人被击杀）。

#### 任务拆分
- [x] T6：扩展 `CombatEnemy`：增加 hp/受击/死亡（并提供销毁/回收入口）。
- [x] T7：实现 `ManualTurretController`：鼠标射线命中敌人，按冷却节奏造成伤害。
- [x] T8：将升级等级绑定到炮台参数（伤害/射速），并在 HUD/结算/Debug 中体现。

### 推荐排期（按依赖顺序）
- 迭代 1（0.5–1 天）：P1-1（能量/金币 HUD + 升级消耗可见）
- 迭代 2（1–2 天）：P1-2（升级实效：基地耐久 + 炮台参数）
- 迭代 3（2–4 天）：P1-3（手动炮台最小可玩 + 敌人血量/死亡链路）
- 打磨（可选 1 天）：命中反馈、冷却 UI、数值微调与防呆

## 6. 验收标准

### 可玩性
- 玩家能够从主菜单进入一局。
- 玩家能够完成工作阶段与休息阶段循环。
- 玩家能够完成至少一波战斗并获得结算。

### 系统性
- 资源变化可追踪。
- 存档可读写（当前：跨局金币已持久化；完整存档待扩展）。
- 配置可调整（波次/敌人等仍以 Inspector 为主，配置表待做）。
- 核心逻辑可测试（部分已有 Editor 测试，待扩大覆盖）。

### 体验性
- 反馈清晰。
- 流程不断裂。
- 首版不会频繁崩溃或卡死。

## 7. 风险点

- 打字与战斗割裂，导致核心循环不连贯。
- 塔防内容扩展过快，拖慢 MVP 进度。
- 数值过早失衡，导致首轮体验过难或过轻。
- 存档与配置格式过早复杂化，增加维护成本。

## 8. 当前建议

- 先按第 1–4 周做出能玩的闭环。
- 第 5–8 周再做成长、UI 和平衡。
- 所有新增内容都优先服务于“能玩一局”的目标。

## 9. 当前执行重点

- **下一步优先**：工作/休息 **双阶段**（打字阶段与战斗阶段切换 + 阶段 UI）；局外 **金币消费**（最小 1～2 个升级项）；波次 **ScriptableObject/JSON 配置** 三选一先做最小版。
- 并行：补 **EditMode** 测试（`ResourceWallet`、`RunSession`、金币持久化、波次进度）。
- 中长期：按第 10 节做 **RunGameLogic** 规则下沉，减少 UI 内写规则。

## 9.1 下一步推荐（按依赖排序）

1. **阶段面板 + 状态**：`Typing` / `Combat` 子状态（或 `Run` 内 FSM），战斗阶段才刷怪、才启用炮台；工作阶段只打字与升级准备。
2. **局外成长 MVP**：主菜单或 Run 前用 **金币** 购买永久 +1 基地初始血或 +1 炮台基础伤害（写回 `PersistentGold` 与单独 PlayerPrefs 键）。
3. **波次配置化**：单份 `WaveConfig`（敌人数、间隔、血量倍率）只读数据，替代 `CombatWaveManager` 硬编码。
4. **打磨**：炮台命中反馈、敌人血条、`AimReticle` 与 Layer 文档化。

## 10. 延后重构需求（基础功能完成后执行）

### 重构目标
- 将胜负判定、波次推进、资源变化等规则统一下沉到纯 C# 逻辑层。
- 将 UI 层收敛为展示与交互转发，不再承载业务规则。
- 明确单一状态源，避免同一局状态在多个 MonoBehaviour 中重复维护。

### 当前痛点（待回收技术债）
- 规则分散在 UI 控制器与战斗适配器中，职责边界不清晰。
- 运行状态存在多处持有与推进，增加了状态不一致风险。
- 部分 UI 流程依赖运行时动态创建，编辑态可视化与协作成本较高。

### 计划方案（第一版）
- 新增统一的 `RunGameLogic`（纯 C#）作为单一状态聚合与规则入口。
- `RunHudController` 仅保留：订阅状态、刷新视图、转发按钮与输入事件。
- `CombatWaveManager` 仅保留 Unity 适配职责：刷怪、对象生命周期、事件上报。
- `GameOverPanel` 与 HUD 采用场景内手动搭建与序列化绑定，减少运行时拼装 UI。

### 拆分任务
- [ ] T1：抽取领域状态模型（运行状态、战斗状态、经济状态）与只读快照。
- [ ] T2：将失败判定与波次推进迁移到纯逻辑层，UI 仅响应事件。
- [ ] T3：移除 UI 层中的规则写入代码，保留展示与输入转发。
- [x] T4：统一 `RunSession` 生命周期与所有权，确保单一状态源。
- [ ] T5：补齐 EditMode 单元测试（GameOver、Retry 重置、波次进度、资源结算）。

### 执行时机与范围控制
- 执行时机：在当前基础玩法闭环、战斗流程、GameOver/Retry 可稳定运行后开始。
- 范围控制：先完成“最小可重构集”（状态源统一 + 失败流程下沉），再做全量迁移。
- 不在本轮重构内处理：美术表现优化、复杂动画系统、跨局成长大改。

### 完成标准
- 规则层不依赖 UnityEngine 命名空间。
- UI 控制器不直接修改核心规则状态。
- 关键流程可通过纯逻辑测试覆盖并稳定通过。
- 场景内 UI 结构在编辑器可见，可直接配置与联调。
